# 六阶段开发流程：Plan-Execute-Test-Fix

> 来源：[Douglas Liles: AI Agent Workflow Orchestration](https://medium.com/@dougliles/ai-agent-workflow-orchestration-d49715b8b5e3)

---

## 概述

Plan-Execute-Test-Fix 是专为软件开发设计的六阶段流程，已被证明将 AI 生成代码的错误减少 60-80%。

```
Clarify → Plan → Execute → Test → Fix → Summarize
   ↑                                    ↓
   └──────── 迭代循环，直到所有测试通过 ←┘
```

---

## 阶段 1：Clarify（澄清）

**目标**：在开始前确保理解需求。

**关键问题**：
```
- 用户想要什么？
- 假设是什么？
- 成功标准是什么？
```

### Clarify 阶段清单

```markdown
## 需求澄清

### 用户需求
[重述用户的需求]

### 我的理解
[对需求的理解]

### 假设
1. [假设 1]
2. [假设 2]

### 需要确认的问题
- [问题 1]
- [问题 2]

### 成功标准
- [标准 1]
- [标准 2]
```

### Clarify 的重要性

> "Without clarification steps, agents misunderstand requirements but proceed confidently." — Douglas Liles

**失败模式**：
```
用户："修改用户模块"
Agent 直接执行 → 创建了错误的文件 → 浪费大量时间回退
```

**正确做法**：
```
用户："修改用户模块"
    ↓
Clarify：请问是指 src/users/ 还是 src/user-service/？
    ↓
用户确认：src/users/
    ↓
继续 Plan 阶段
```

---

## 阶段 2：Plan（规划）

**目标**：在执行前规划具体步骤。

**关键输出**：
```
- 步骤列表
- 识别的风险
- 需要的工具和文件
- 预期的中间结果
```

### Plan 格式

```markdown
## 执行计划

### 步骤 1: [操作]
- 工具：[使用的工具]
- 文件：[涉及的文件]
- 预期输出：[预期结果]

### 步骤 2: [操作]
- ...

### 识别的风险
- [风险 1]：影响 [影响描述]，缓解措施 [缓解措施]
- [风险 2]：...

### 回滚计划
如果步骤 X 失败，我将 [回滚操作]
```

### Plan 的门控

```
Plan 使用了不允许的工具？
    ↓ 是
拒绝 → 请求修复 → 重新规划
    ↓ 否

Plan 缺少停止条件？
    ↓ 是
拒绝 → 添加停止条件 → 重新规划
    ↓ 否

进入 Execute 阶段
```

---

## 阶段 3：Execute（执行）

**目标**：实现小而孤立的变更。

**执行原则**：

| 原则 | 说明 |
|------|------|
| **小变更** | 每个变更应可在单个 PR 中审查 |
| **孤立** | 变更不应影响无关模块 |
| **可回滚** | 每个变更都应有回滚计划 |

### Execute 的最佳实践

```
变更 1（文件 A）→ 测试 → 通过？
    ↓ 是
变更 2（文件 B）→ 测试 → 通过？
    ↓ 是
变更 3（文件 C）→ 测试 → 通过？
    ↓ ...
```

**禁止模式**：
```markdown
❌ 一次性修改 20 个文件
❌ 边改边测试，最后一次性运行完整测试
❌ 实现功能后再写测试
```

---

## 阶段 4：Test（测试）

**目标**：在继续下一步前验证当前变更。

**核心原则**：
> "No subsequent step executes until current tests pass."

### 测试类型

| 类型 | 运行时间 | 运行频率 |
|------|----------|----------|
| 单元测试 | < 5s | 每次变更后 |
| 类型检查 | < 10s | 每次变更后 |
| Lint | < 5s | 每次变更后 |
| 集成测试 | < 60s | 每次 PR |
| E2E 测试 | < 5min | 每次 PR |

### Test 的门控

```
测试通过？
    ↓ 否
Fix 阶段 → 修复问题
    ↓ 是
下一个变更
```

---

## 阶段 5：Fix（修复）

**目标**：对失败测试进行修复。

**修复原则**：

| 原则 | 说明 |
|------|------|
| **显式推理** | 说明为什么出问题 |
| **针对性修复** | 不要为了修一个问题引入新问题 |
| **防止重试循环** | 盲目重试不会改善结果 |

### Fix 的常见模式

```typescript
// 失败分析模板
测试失败："Component test failed"
  ├─ 原因分析：
  │   ├─ 假设 1：组件逻辑错误 → 验证：检查组件源码 → 不是
  │   ├─ 假设 2：测试断言错误 → 验证：检查测试 → 是！
  │   └─ 假设 3：依赖 mock 错误 → 验证：检查 mock → 不是
  └─ 结论：测试断言与实际行为不符
      ├─ 实际行为：[描述]
      ├─ 期望行为：[描述]
      └─ 修复方案：更新测试断言以匹配实际行为
```

### 盲目重试 vs 显式修复

```
❌ 盲目重试（无效）：
测试失败 → 修改代码 → 测试仍失败 → 修改代码 → 测试仍失败 → ...

✅ 显式修复（有效）：
测试失败 → 分析原因 → 识别根因 → 针对性修复 → 测试通过
```

---

## 阶段 6：Summarize（总结）

**目标**：记录变更和学习。

**总结内容**：

```markdown
## 总结

### 做了什么
[简要描述]

### 变更的文件
- [文件 1]
- [文件 2]

### 验证方式
- [测试 1]：通过
- [测试 2]：通过

### 学到什么
- [教训 1]
- [教训 2]

### 剩余工作（如有）
- [待办 1]
- [待办 2]
```

## 完整循环

```
Clarify ───────────────────────────────────────┐
    ↓                                          │
Plan ─────────────────────────────────────────┤
    ↓                                          │
Execute ──────────────────────────────────────┤
    ↓                                          │
Test ─────────────────────────────────────────┤
    ↓ 通过                                      │ 失败
Next ─────────────────────────────────────────┤
    ↓                                          │
... ─────────────────────────────────────────┤
    ↓                                          │
Summarize ────────────────────────────────────┘
```

---

## 对 cc-harness 的映射

| 阶段 | cc-harness 实现 |
|------|----------------|
| Clarify | SKILL.md 的"触发场景" + 触发前确认 |
| Plan | Skill 的"工作流"章节定义步骤 |
| Execute | Skill 执行具体操作 |
| Test | `evals/` 测试 + Lint + 类型检查 |
| Fix | 测试失败时的修复流程 |
| Summarize | Git commit message + 进度文件更新 |

---

## 与七阶段模型（MEV）的对比

| Plan-Execute-Test-Fix | MEV 七阶段 | 侧重点 |
|----------------------|-----------|--------|
| Clarify | Intake + Triage | 需求理解 |
| Plan | Plan | 步骤规划 |
| Execute | Execute | 实现 |
| Test | Validate | 验证 |
| Fix | （嵌入 Test 失败流）| 修复 |
| Summarize | Audit | 记录 |

两个模型可以结合使用：Clarify/Plan/Execute/Test/Fix/Summarize 作为内循环，Intake/Triage/Validate/Commit 作为外层治理。
