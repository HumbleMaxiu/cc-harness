# Product Sense — cc-harness

## 产品为谁服务

### 用户画像

| Persona | 需求 |
|---------|------|
| **AI-First 开发者** | 在项目中引入结构化的 AI 协作流程，提升代码质量和一致性 |
| **Harness 新用户** | 不知道什么是 harness，希望快速起步而非阅读大量文档 |
| **AI 工具探索者** | 已经在用 Cursor/Windsurf/Codex，希望迁移或整合 |
| **团队 lead** | 为团队统一 AI 工具标准，建立交接规范 |

## 产品目标

`cc-harness` 不是零散 skill、agent 或 hooks 的集合，而是一个**以文档为核心的 AI 协作系统**。

我们希望提供的是：

- 一套围绕文档运行的 harness 能力
- 让 skills、agents、hooks、MCPs、commands、rules 都能参与文档的读取、写入、补齐、同步和最终收口
- 让项目中的关键 docs 不只是“被生成”，而是始终有明确的使用场景、维护路径和消费入口

换句话说，文档不是附属产物，而是整个 harness 的操作界面、长期记忆和协作事实源。

### 当前阶段重点

在基础阶段，优先补齐这几类能力：

1. **自然语言触发的收口**
   - 例如 `/doc-sync`、`/feedback`、`/harness-quality-gate`
2. **流程型收口**
   - 例如 `/dev-workflow` 中多角色协作对计划、验证、反馈和文档的收口
3. **scaffold 型收口**
   - 例如 `/harness-setup` 读取项目事实、理解用户需求、套用模板后生成 `docs/`、`AGENTS.md`、`ARCHITECTURE.md`
4. **hook 型收口**
   - 在会话生命周期内持续把 plan、memory、feedback 和 drift signals 带回上下文

### 后续扩展方向

当这些文档中心化的基础能力稳定后，再接入更强的工具型能力，例如：

- UI 还原度检测
- e2e 测试生成
- 测试与验证工具

这些能力的目标不是替代文档中心化，而是建立在文档收口稳定之后，进一步补全 harness 的验证和执行能力。

### 非目标（Non-goals）

- ❌ 不提供运行时执行引擎（依赖 Claude Code 环境）
- ❌ 不提供 UI 或 Web 界面
- ❌ 不强制特定的编程语言或框架

### UX 原则

1. **越少越好** — Scaffold 生成的文件数最小化，但每个都有明确目的。
2. **渐进式披露** — 用户先看到 AGENTS.md 的简短索引；需要时再深入 docs/。
3. **上下文自解释** — 每个文档文件的开头有一句话描述其用途。
4. **跨平台兼容** — Bridge 文件支持多种 agent IDE，最大化适用范围。

### 什么是"好"的 harness

- 新开发者 clone 后，AI agent 能独立完成端到端任务
- 所有设计决策和执行状态都在 Git 中，无需外部工具或记忆
- Agent 工作流有记录（交接文档），人类可以审查和接管
- 每一类关键 docs 都有明确的读取入口、维护入口和实际使用场景，而不是只生成不消费
