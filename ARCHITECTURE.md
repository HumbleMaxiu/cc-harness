# Architecture — cc-harness

`cc-harness` is a source-first harness repository. It stores reusable skills, hook scripts, install logic, and documentation. Host-specific runtime folders are generated by installation and are not checked in.

## Source Tree

```text
cc-harness/
├── AGENTS.md
├── README.md
├── install.sh
├── package.json
├── skills/
│   ├── dev-workflow/

│   ├── architect/

│   ├── reviewer/
│   ├── tester/
│   └── ...
├── scripts/
│   ├── install.mjs
│   ├── hooks/
```

## Runtime Installation

The installer projects the source tree into a host runtime:

| Host | Generated In Target Project | Source Used |
|------|-----------------------------|-------------|
| Claude Code | `.claude/skills`, `.claude/scripts/hooks`, `.claude/settings.json` | `skills/`, `scripts/hooks/` |
| Codex | `.codex/skills`, `.codex/scripts/hooks`, `.codex/config.toml`, `.codex/hooks.json` | `skills/`, `scripts/hooks/` |

These generated directories are install artifacts. They do not define the repository's source model.

## Role Model

Former standalone role definitions are now ordinary skills:

- `/architect`
- `/challenger`
- `/developer`
- `/reviewer`
- `/tester`
- `/feedback-curator`

`/dev-workflow` coordinates these role skills by contract. It may ask the host to execute or delegate a role skill, but it no longer depends on host-specific role definition files.

## Hook Model

`scripts/hooks/` contains the shared hook runtime. The installer writes host-specific hook configuration:

- Claude Code hooks use `.claude/settings.json`
- Codex hooks use `.codex/hooks.json` and require `codex_hooks = true`

The hook scripts remain fail-open: if payload parsing, path resolution, or plan lookup fails, they avoid blocking the host and log useful context where configured.

## Validation

This source tree currently has no repo-local test/check script. Validate installation behavior by running `./install.sh --target both --dest <target-project>` and inspecting the generated Claude Code / Codex runtime files.
