# Architecture Decision Records (ADR) 模板

> ADR 用于记录重要的架构决策及其上下文和后果

---

## ADR 文件命名规范

```
YYYY-MM-DD-[简短描述].md
例：2026-04-09-use-typescript.md
```

## 模板结构

```markdown
# ADR-[编号]: [决策标题]

> 日期：YYYY-MM-DD
> 状态：[提议 | 已接受 | 已废弃 | 已替换]
> 决策者：[团队/个人]

## 背景

[描述做出此决策的场景和动机]
- 面临的问题是什么？
- 有什么约束条件？
- 利益相关者是谁？

## 决策

[描述最终做出的决策]

### 选项对比

| 选项 | 优点 | 缺点 | 成本 |
|------|------|------|------|
| 选项 A | ... | ... | ... |
| 选项 B | ... | ... | ... |
| 选项 C | ... | ... | ... |

### 选择理由
[解释为什么选择此方案]

## 后果

### 正面
- [可衡量的好处]

### 负面
- [已知的权衡或限制]

### 风险缓解
- [如何降低负面影响]

## 相关决策

- [相关 ADR 链接]
```

## ADR 示例

### 示例：选择数据库

```markdown
# ADR-001: 使用 PostgreSQL 作为主数据库

> 日期：2026-04-09
> 状态：已接受
> 决策者：后端团队

## 背景

我们需要为新项目选择主数据库。团队有 5 人，预计处理 10 万 DAU。

## 决策

使用 **PostgreSQL 16** 作为主数据库，配合 Prisma ORM。

### 选项对比

| 选项 | 优点 | 缺点 | 成本 |
|------|------|------|------|
| PostgreSQL | 成熟、JSON 支持、扩展性 | 需要运维 | 中 |
| MongoDB | 灵活、Schema-less | 事务支持弱 | 中 |
| MySQL | 广泛使用 | JSON 支持差 | 低 |

### 选择理由
- 团队熟悉 PostgreSQL
- 强事务支持满足金融场景
- 生态成熟，有 RDS 托管

## 后果

### 正面
- 减少学习曲线
- 支持复杂查询
- 有托管服务降低运维负担

### 负面
- 水平扩展相对复杂
- 需要定期维护（升级、备份）

### 风险缓解
- 使用 RDS 自动备份
- 设置监控告警
```

---

## ADR 维护原则

1. **每个重要决策都需要 ADR**
2. **在决策时记录，而非事后补录**
3. **状态变化时更新文件头**
4. **定期回顾，清理过时决策**
5. **ADR 编号按项目顺序递增**

## 决策状态说明

| 状态 | 含义 |
|------|------|
| 提议 | 正在讨论中 |
| 已接受 | 已决定并执行 |
| 已废弃 | 不再使用 |
| 已替换 | 被新 ADR 替代 |
