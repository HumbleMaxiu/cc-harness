# 渐进式改进现有 AGENTS.md

如何评估和逐步改进已有的 AGENTS.md 文件。

## 评估现有 AGENTS.md

### 质量评分矩阵

| 维度 | 0 分 | 1 分 | 2 分 | 3 分 |
|------|------|------|------|------|
| **完整性** | 无文件 | 有但缺失核心章节 | 核心章节全但无细节 | 完整且适度详细 |
| **行数** | >500 | 300-500 | 200-300 | 50-200 |
| **时效性** | 未更新 >1 年 | 未更新 >6 月 | 未更新 >1 月 | 最近 30 天 |
| **可操作性** | 模糊规则 | 部分可操作 | 大部分可操作 | 全部可操作 |
| **中文约束** | 缺失 | 部分 | 存在但弱 | 强制且明确 |

**总分** = 各维度之和，评估结果：
- 0-5 分：需要重写
- 6-10 分：需要大幅改进
- 11-13 分：需要小幅改进
- 14-15 分：优秀

## 常见问题与修复

### 问题 1：指令膨胀（>300 行）

**症状**：AGENTS.md 超过 300 行，Agent 出现 "lost in the middle"

**原因**：把 AGENTS.md 当成百科全书

**修复**：
1. 将详细约定拆分到 `docs/` 目录
2. AGENTS.md 只保留"路由"——指向具体文档
3. 保留 50-200 行

**修复示例**：

```markdown
# 项目名

## 关键约定
1. 所有 API 请求必须使用 `src/api/client.ts` 封装
2. [保持 3-5 条最重要规则]

## 详细约定

> 编码风格详见 `docs/conventions.md`
> 测试规范详见 `docs/testing.md`
> 部署流程详见 `docs/deployment.md`
```

---

### 问题 2：过时时效

**症状**：AGENTS.md 描述与实际代码不符

**修复**：
1. 标记最后更新时间
2. 批量验证关键约定是否仍然正确
3. 移除已过时的规则

**修复示例**：

```markdown
# 项目名

> [最后验证: 2024-01-15]
```

---

### 问题 3：缺少中文约束

**症状**：根目录 AGENTS.md 没有要求中文输出

**修复**：在"关键约定"或单独章节添加：

```markdown
## 语言约束

**必须使用中文输出**。所有回复、注释、提交信息均使用中文。
```

---

### 问题 4：假约定（模型已知的规则）

**症状**：包含大量"模型本来就会"的规则

**示例**：
```markdown
❌ 不要使用 var，要用 const/let
❌ 不要硬编码敏感信息
❌ 不要提交密码到 Git
```

**修复**：删除或仅保留**真正特殊**的约定。

---

### 问题 5：缺少文档地图

**症状**：有 `docs/` 目录但 AGENTS.md 没有引用

**修复**：添加文档地图章节：

```markdown
## 文档地图

| 主题 | 文件 |
|------|------|
| 架构设计 | `docs/architecture.md` |
| API 约定 | `docs/api.md` |
| 编码规范 | `docs/conventions.md` |
| 测试策略 | `docs/testing.md` |
```

---

### 问题 6：空项目（无 AGENTS.md）

**症状**：项目已有代码但无 AGENTS.md

**修复**：按场景 1 生成新的 AGENTS.md

---

## 改进优先级

按以下顺序改进，一次一个层级：

### P0（必须修复）

1. ✅ 缺少中文输出约束
2. ✅ 包含过时错误信息（比没有更糟）
3. ✅ 缺失核心章节（项目概述、快速开始）

### P1（强烈建议）

1. 行数 >300 的压缩到 200 以内
2. 添加文档地图
3. 补充关键约定（3-5 条具体可操作规则）

### P2（可选优化）

1. 添加最后验证日期
2. 添加子模块索引
3. 优化语言风格（更命令式）

---

## 改进流程

### 步骤 1：诊断

```
1. 读取现有 AGENTS.md
2. 对照质量评分矩阵打分
3. 识别 top 3 问题
4. 确定改进优先级
```

### 步骤 2：制定改进计划

```markdown
## AGENTS.md 改进计划

### P0 问题（必须修复）
- [ ] 添加中文输出约束

### P1 问题（建议修复）
- [ ] 将详细约定拆分到 docs/
- [ ] 添加文档地图

### P2 问题（可选）
- [ ] 添加最后验证日期
```

### 步骤 3：执行改进

每次改进后验证：
- [ ] 行数符合目标
- [ ] 关键章节完整
- [ ] Agent 能正常加载和使用

### 步骤 4：用户确认

向用户展示改进内容：
```
已完成以下改进：

### P0
- 添加中文输出约束

### P1
- 将 150 行详细约定拆分到 `docs/conventions.md`
- 添加文档地图，指向 4 个 docs/ 文件

### 改进前
- 行数：450+
- 文档地图：无

### 改进后
- 行数：180
- 文档地图：4 个文件
```

## 渐进改进 vs 重写

| 情况 | 选择 |
|------|------|
| 有基本结构，需要补充 | 渐进改进 |
| 结构混乱，内容矛盾 | 重写 |
| >80% 内容需要改 | 重写 |
| 时间紧迫 | 渐进改进 |
| 想保留历史上下文 | 渐进改进 |
