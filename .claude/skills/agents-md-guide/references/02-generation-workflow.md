# AGENTS.md 生成详细流程

从用户需求到完整 AGENTS.md 的逐步生成流程。

## Phase 1：需求澄清

如果用户提供了模糊需求，先澄清：

### 必问问题

| 问题 | 作用 |
|------|------|
| 项目类型是什么？ | 决定模板变体（Web/CLI/SDK/脚本） |
| 核心技术栈？ | 确定语言特定约定 |
| 有现有代码吗？ | 区分"全新项目"和"已有项目" |
| 团队规模？ | 决定约定数量和复杂度 |
| Agent 主要用途？ | 编码 / Review / 文档 / 全流程 |

### 需求类型判断

**模糊需求示例**：
- "帮我设置这个项目"
- "让它能用 AI Agent"
- "创建 AGENTS.md"

**清晰需求示例**：
- "为这个 React + Node.js 的电商后端创建 AGENTS.md"
- "为 `packages/auth` 子模块创建 AGENTS.md"

## Phase 2：信息收集

### 2.1 读取现有文件

如果项目已有代码，读取关键文件了解项目：

```
1. package.json / Cargo.toml / go.mod 等依赖文件
2. 根目录 README.md
3. src/ 目录结构
4. 现有配置文件 (.eslintrc, tsconfig.json 等)
```

### 2.2 构建信息清单

```
项目基本信息
├── 项目名称：
├── 项目类型：
├── 技术栈：
├── 团队规模：
└── Agent 用途：

缺失信息（需要用户补充）
├── [ ] 项目的核心约束？
├── [ ] 必须遵守的规范？
└── [ ] 特殊上下文？
```

### 2.3 关键约定收集（必须执行）

> ⚠️ **关键约定不能凭空编造。必须从用户获取，或从现有代码推断。**

在收集完基本信息后，向用户提出以下问题：

| 问题 | 作用 | 无回答时的默认策略 |
|------|------|------------------|
| 项目有安全要求吗？（认证、数据加密、隐私） | 生成安全相关约定 | 从代码推断 |
| 测试覆盖率有要求吗？ | 生成对应测试规范 | 从现有测试文件推断 |
| 有命名规范吗？ | 生成命名约定 | 从代码推断 |
| 有什么绝对不能做的事？ | 生成禁止条款 | 添加通用禁止项 |
| 有特殊的 API 风格或协议约定吗？ | 生成 API 约定 | 从现有代码推断 |
| 需要约束 Agent 回复语言吗？（中文/英文） | 生成语言约束条款 | 不添加（按需） |

**规则**：
- 用户提供的约定 > 从代码推断的约定 > 不生成
- 关键约定不超过 10 条
- 代码推断只用于填充用户未覆盖的空白

### Phase 2.5：关键约定收集 ⚠️（必须执行）

**关键约定不能只靠代码推断——必须从用户获取。**

如果用户已提供关键约定 → 直接使用。

如果用户未提供，按以下顺序询问：

| 问题 | 作用 |
|------|------|
| 项目有安全要求吗？（认证、数据加密） | 影响安全相关约定 |
| 测试覆盖率要求？ | 生成对应测试规范 |
| 有命名规范吗？ | 生成命名约定 |
| 有什么绝对不能做的事？ | 生成"禁止"条款 |
| 有特殊的 API 风格或协议约定吗？ | 生成 API 约定 |

**规则**：
- 用户确认优先：用户提供的约定直接使用，不替换
- 代码推断为补充：仅用于填充用户未覆盖的空白
- 禁止凭空编造：如果代码中没有明确模式，不生成对应约定
- 条数控制：5-10 条，不超过 10 条

## Phase 3：生成 AGENTS.md

### 步骤 1：选择模板

| 场景 | 模板选择 |
|------|----------|
| 全新空项目 | 使用 `01-agents-template.md` 基础模板 |
| 已有项目 | 基于模板 + 从代码推断内容 |
| 子目录模块 | 使用子目录模板 |
| 已有 AGENTS.md | 进入改进流程 |

### 步骤 2：填充内容

**必填章节**（根目录）：
1. Project Overview（1-2 句话）
2. Quick Start（3 条核心命令）
3. Directory Structure（关键目录 + 一句话描述）
4. Key Conventions（**来源：用户确认 > 代码推断**，5-10 条）
5. Documentation Map（指向 `docs/` 文件，**禁止**内部开发路径）
6. **Language Constraint**（可选，仅在用户明确要求时添加）

**可选章节**（按需添加）：
- Tech Stack 详情
- Code Style
- Common Tasks 命令
- Submodules 索引

**可选章节**（按需添加）：
- 技术栈详情
- 代码风格
- 常见任务命令
- Submodules 索引

### 步骤 3：质量检查

对照清单自检：
- [ ] 行数 ≤ 200 行
- [ ] 必填章节完整
- [ ] 中文输出约束存在
- [ ] 无过时信息
- [ ] 无百科全书式内容

## Phase 4：用户确认

生成后向用户展示，重点让用户审核**关键约定**部分：

```
已生成 AGENTS.md，包含以下内容：

## 项目概述
[1-2 句话]

## 关键约定 ← 请务必审核此处
1. [约定 1 — 来源：用户提供 / 代码推断]
2. [约定 2 — 来源：代码推断]
...

## 文档地图
[表格]

请确认以下内容：
1. "关键约定"中的每一条是否符合你的预期？
2. 有没有遗漏的约定？（比如安全、API 风格）
3. "文档地图"中的文档路径是否正确？

如需调整，请告诉我具体修改内容。
```

## 生成示例

### 输入
```
用户："为这个 TypeScript + React 项目创建 AGENTS.md"
```

### 读取项目后推断
- 项目类型：Web 应用
- 技术栈：TypeScript, React, Vite
- 测试：Vitest + React Testing Library
- Lint：ESLint + Prettier

### 输出

```markdown
# 我的项目

> React + TypeScript Web 应用

## 快速开始

```bash
npm install
npm run dev
npm test
```

## 技术栈

- **前端框架**: React 18 + TypeScript
- **构建工具**: Vite
- **测试**: Vitest + React Testing Library
- **代码检查**: ESLint + Prettier

## 目录结构

```
src/
├── components/    # 可复用 UI 组件
├── pages/         # 页面组件（路由级别）
├── hooks/         # 自定义 React Hooks
├── utils/         # 纯工具函数
└── __tests__/     # 集成测试
```

## 关键约定

1. **组件放在 `src/components/`**，页面组件放在 `src/pages/`
2. **每个组件必须有对应测试**，文件名 `ComponentName.test.tsx`
3. **使用 CSS Modules**，禁止全局 CSS（除 `index.css`）
4. **API 调用封装在 `src/api/`**，禁止在组件内直接 fetch

## 中文输出约束

**必须使用中文输出**。所有回复、注释、提交信息均使用中文。
```
